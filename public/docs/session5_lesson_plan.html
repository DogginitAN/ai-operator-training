What This Session Is About

"Agent" has become a buzzword. Everyone says they're building agents. Most of them are building pipelines with LLM steps. That's not a bad thing — most production systems should be pipelines. But knowing the difference is what separates someone who can build AI-powered tools from someone who can architect AI-native systems.

This session makes the difference visceral. You're going to run the same problem — Provider Health Outreach for HouseAccount — through both architectures. Same data, same tools, same goal. The outputs will be different. And understanding why they're different is the whole lesson.

0–20 min Discover: What You Already Built Is a Pipeline

Start with what Matt knows. Walk through the HouseAccount chatbot — the intake flow he built and demoed in Session 3.

Ask the question: "At each step, who decided what happens next — you or Claude?"

  • Customer enters a request → Claude interprets it (LLM step) → but the system always goes to fuzzy match next (you decided)
  • Fuzzy match finds a rate book item → system always goes to provider selection next (you decided)
  • Provider selected → system always goes to pricing next (you decided)
  • Pricing presented → system always goes to negotiation next (you decided)
  • Customer accepts → system always goes to booking creation (you decided)

Every step is predetermined. Claude is smart within each step — it interprets natural language, does fuzzy matching, handles negotiation. But it never chooses the next step. You hardcoded the sequence. That's a pipeline.

A pipeline is a system where you decide the control flow and the LLM fills in the blanks. An agent is a system where the LLM decides what to do next from a set of available tools. The distinction is one question: who's routing?

This isn't a criticism. Pipelines are good. They're predictable, debuggable, and fast. Most production systems should be pipelines. The point is to see it clearly before we show the alternative.

20–40 min Invert: Run the Same Problem Both Ways

Now we run the Provider Health system. Same data, same goal, two architectures. You already pulled the repo — the materials are ready.

Run the pipeline first. Open Claude Code in session-05-agentics/. Tell it:
Follow the instructions in pipeline.md. Process every provider in sample_data/providers.json through the fixed sequence. Save the report to output/pipeline_report.md.
Watch it work. Every provider goes through the same six steps. Same formula, same thresholds, same template.
Then run the agent. In a fresh Claude Code session (same directory):
Follow the instructions in agent.md. For each provider in sample_data/providers.json, assess the situation and decide what action to take using the tools in tools.md. Save the report to output/agent_report.md.
Watch it work. This time, notice which tools it picks for each provider. Notice the order. Notice when it skips tools entirely.

What you'll see:

  • The pipeline treats every provider the same. Eight providers, six steps each, identical structure.
  • The agent treats each provider differently. It might run two tools on a healthy provider and five on a troubled one. It might escalate one and update the database for another. The model is routing.
The moment to watch for: When the agent does something the pipeline wouldn't — skips a step, combines two issues, catches a nuance in the notes field, or decides to escalate instead of outreach. That's the model making a judgment call. That's what makes it an agent.
40–60 min Observe: Read the Reports Side by Side

Open both reports — output/pipeline_report.md and output/agent_report.md. Compare them provider by provider.

Questions to discuss:

  • Which providers got the same result from both systems? Those are the cases where the pipeline is sufficient — the fixed sequence arrived at the right answer.
  • Which providers got different results? What did the agent see that the pipeline missed?
  • Did the agent ever take an action the pipeline couldn't — like escalating, recommending a database update, or deciding no action was needed despite a mediocre health score?
  • Did the pipeline ever produce a result that feels wrong? A message sent when it shouldn't have been, or a problem categorized too simply?

Walk through the interesting providers:

  • Mike's Plumbing (P-1002) — pricing AND geography problems interacting. The pipeline picks the most frequent rejection reason. The agent should see they're connected.
  • RedLine (P-1004) — previous outreach went unanswered. The pipeline checks cooldown and skips. The agent should recognize unanswered outreach means escalation, not silence.
  • AllPro (P-1006) — possibly disengaging entirely. A health score doesn't capture "this provider might be leaving." The agent should consider escalation.
  • QuickFix (P-1008) — overextended, not unhealthy. Schedule conflicts aren't a sign of a bad provider, they're a sign of a popular one. The agent should handle this differently than the pipeline.
The agent's advantage is judgment at the routing level. The LLM isn't just better at writing the outreach message — it's better at deciding whether outreach is even the right action. That's a fundamentally different capability than filling in a template.
60–75 min Collapse: The Agent Discovers the Pipeline

Here's the part most people miss. Look at the agent report again. How many providers did the agent handle the same way?

If you ran this agent on 100 providers instead of 8, you'd start seeing patterns. The agent would handle pricing problems the same way 80% of the time. Geography mismatches would follow a pattern. Healthy providers would get the same "no action needed" assessment.

The agent discovers pipelines. Once you can see the pattern — once the problem becomes legible — you pipeline it. The agent handles the 10-20% of cases that don't fit the pattern. Over time, the agent surface area shrinks as you codify more of its decisions into fixed sequences.

The lifecycle:

  • Phase 1: You don't know the right sequence. Deploy an agent. Let it explore.
  • Phase 2: Watch the logs. See which tool sequences repeat. Those are your candidate pipelines.
  • Phase 3: Pipeline the common paths. Keep the agent for edge cases and novel situations.
  • Phase 4: The agent handles less and less over time. Most of the system is now deterministic pipelines that are fast, cheap, and predictable.

Apply this to HouseAccount:

  • The chatbot intake flow you built? That's Phase 4. You knew the sequence. It's a pipeline. Correct.
  • The Provider Health system? Today it's Phase 1. You don't yet know the best outreach strategy for every scenario. An agent explores the space.
  • Six months from now, with logs from hundreds of providers? You'll pipeline 80% of it and keep an agent for the weird cases.
The capital-T truth about agents: They're a discovery tool for finding the pipeline you should have built. The goal isn't to have an agent running forever. The goal is to use the agent to make the problem legible, then collapse it into something deterministic. The best AI systems are mostly pipelines with agent layers where the problem is still being figured out.
75–90 min Apply: Where Do Agents Belong in HouseAccount?

Now that the distinction is clear, map it to the real business. Walk through the four agents Matt originally scoped and ask: is this actually an agent, or is it a pipeline with the agent label?

  • Provider Sales Agent — Outreach to potential providers. Is the sequence known? (Probably yes → pipeline with LLM-powered personalization.)
  • Customer Intake Agent — Job scoping and matching. Is the sequence known? (Mostly yes → pipeline. But edge cases like multi-trade jobs or unusual requests might need agent routing.)
  • Pricing Estimate Agent — Rate book matching and quote generation. (Pipeline. The logic is deterministic with LLM-powered fuzzy matching.)
  • Concierge Agent — The orchestrator. This is the one that might be a real agent — it needs to decide: does this homeowner need pricing help? A different provider? An escalation? The routing is the value.

Homework assignment and looking ahead:

  • Connect to real data. Swap sample_data/providers.json with a Metabase export of your actual providers. Run the agent on real data. What does it find?
  • Map your HouseAccount systems. For every AI-powered thing you've built or want to build, write down: is the control flow decided by me or by the model? Label each one "pipeline" or "agent." Most will be pipelines. That's correct.
  • Run the agent 3 more times on different subsets. Look for the repeating patterns. Which tool sequences show up over and over? Those are your candidate pipelines to collapse.
  • Pick one thing that should be an agent. Something at HouseAccount where you don't yet know the right sequence — where the problem isn't legible yet. Write a CLAUDE.md and tools.md for it. Don't pipeline it. Let it explore.
REF Quick Reference

The One-Line Distinction:

Pipeline = you decide the control flow. Agent = the model decides the control flow.

The Agent-to-Pipeline Lifecycle:

  • Phase 1: Problem is ambiguous → deploy an agent to explore
  • Phase 2: Watch logs → identify repeating tool sequences
  • Phase 3: Pipeline the common paths → agent handles edge cases
  • Phase 4: Most of the system is deterministic → agent surface area minimal

When to Use Which:

  • Pipeline when: You know the sequence. The steps are predictable. Speed and cost matter. You want consistency.
  • Agent when: You don't know the right sequence yet. The problem has branching paths. Different inputs need fundamentally different handling. You need discovery before optimization.

The HouseAccount Agent Map:

  • Customer Intake → Pipeline (with agent layer for edge cases)
  • Pricing Estimate → Pipeline
  • Provider Sales → Pipeline with LLM personalization
  • Provider Health → Agent (discovery phase) → will collapse into pipeline over time
  • Concierge → Agent (routing is the value)
