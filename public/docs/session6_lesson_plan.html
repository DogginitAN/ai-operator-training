What This Session Is About

Two things that are bottlenecking you right now: expertise on demand and managing multiple workstreams. Skills solve the first — they plug domain knowledge (UX design, code review, data analysis) into Claude so it stops being a generalist and becomes a specialist when you need it. The workbench solves the second — setting up your terminal so you can manage multiple projects, multiple Claude Code sessions, and multiple contexts without drowning in tabs.

Together they answer the question you asked last session: "How do I manage all of this at once?"

0–15 min What's a Skill, Really?

Check-in: what happened since last session? What did you build? Where did the agent vs. pipeline distinction show up in your work?

Then the concept: you mentioned that you downloaded a UX design skill from Reddit, referenced it in Chat, and it helped — but you couldn't figure out how to make Claude Code use it when executing. That gap is what we're closing today.

A skill is an MD file that gives Claude domain expertise. Claude is a generalist — it knows a little about UX, code review, data analysis, marketplace operations. A skill makes it a specialist by giving it: a role to play, principles to follow, a process to execute, and an output format to hit. Same Claude, dramatically different results.

The anatomy of a skill:

  • Role — who Claude is acting as (senior product designer, security engineer, data analyst)
  • Principles — the standards to apply (accessibility, performance, mobile-first)
  • Process — the steps to follow (review checklist, analysis framework)
  • Output format — what the result looks like (structured report, annotated code)

Where skills live:

  • Chat project — add the skill MD to your project instructions or upload it as a project file. Every conversation in that project gets the expertise.
  • Claude Code — put the skill in your project directory. Reference it in CLAUDE.md: When doing UI work, follow skills/ux-design.md. Claude Code reads CLAUDE.md on launch.
  • Both — if you want the same expertise in Chat and Code, install in both. They don't share brain.
The gap Matt hit: A skill in Chat doesn't automatically apply in Code. They're separate contexts. Last session's mantra: "two different brains." The skill needs to exist in both places if you want both brains to have the expertise.
15–35 min Install & Run: UX Design Skill Live

We'll install the UX design skill from the toolkit repo and test it on something real from HouseAccount.

Step 1: Open the skills/ux-design.md file from session-06-workbench/. Read through it together — role, principles, review process, output format. This is what an expert designer thinks about that you're not thinking about.
Step 2: Install it in Claude Code. Copy the skill file into your HouseAccount project directory. Add a line to your CLAUDE.md: When reviewing or building UI, follow the design principles in skills/ux-design.md
Step 3: Test it. Open Claude Code in your HouseAccount project. Ask it to review one of your existing pages — the customer intake chat, the admin dashboard, whatever feels roughest. Compare the feedback you get now vs. what Claude would have said without the skill.

What to notice: Without the skill, Claude gives generic feedback — "looks good, maybe add some padding." With the skill, it runs a structured review: first impression, information architecture, visual design, interaction design, edge cases. Same brain, different expertise plugged in.

Where to find more skills: GitHub (search "claude skill md"), Reddit (r/ClaudeAI), or build your own. The best skills come from your domain — nobody has written the "HouseAccount provider evaluation" skill except you. We'll talk about building custom skills in homework.
35–55 min Build Your Workbench

You said the switching cost between projects is killing you. Let's fix that. Windows Terminal has everything you need — you just need to know the shortcuts and the mental model.

Think of your terminal as a cockpit. Tabs are projects or functions — one for HouseAccount, one for Provider Health, one for Git. Panes are views within a project — Claude Code on the left, dev server on the right. Rename every tab so you always know where you are.

The shortcuts that matter:

  • Alt+Shift+D — split pane vertical (side by side)
  • Alt+Shift+- — split pane horizontal (top/bottom)
  • Alt+Arrow — navigate between panes
  • Alt+Shift+Arrow — resize pane
  • Ctrl+Alt+[1-9] — jump to tab by number
  • Right-click tab → Rename
Build your layout live. We'll set up a workspace together:
  • Tab 1: "HouseAccount" — Claude Code left, dev server right
  • Tab 2: "Provider Health" — Claude Code full width
  • Tab 3: "Git" — staging, pushes, conflict resolution
  • Tab 4: "Toolkit" — skills, research, session materials
Practice switching between tabs and panes until the shortcuts are in your fingers.

This is your answer to the question "how do I run multiple threads on the same project?" — each thread gets a pane or a tab. They're all in front of you. No switching cost because you never leave the terminal.

55–65 min The Startup Script

Setting up your workspace manually every morning takes 5 minutes and 20 clicks. A startup script does it in one command.

Open startup-scripts/houseaccount-workspace.ps1 from the toolkit. Walk through it together — each line opens a tab, names it, cd's to the right directory, and optionally splits panes. Customize the paths for your machine. Run it and watch your full workspace appear.

The unlock: You can add this to your PowerShell profile so it runs automatically when you open Terminal, or create a shortcut on your desktop. One click, full cockpit, ready to work.

This is the tmux equivalent for Windows. tmux has session persistence — you can detach and reattach. Windows Terminal doesn't do that natively. The startup script is your workaround — it doesn't restore state, but it recreates your workspace in seconds. For now, that's enough.
