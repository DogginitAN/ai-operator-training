10–25 min Claude Code Deep Dive

Goal: Get him fully comfortable in the terminal before building anything. He needs to feel confident launching, navigating, and controlling Claude Code.

Last session you saw Claude Code briefly. Today we're going to live in it. This is the tool that turns 'I wish we had a tool that does X' into a working prototype in minutes. But first, let's make sure you're comfortable with the controls.
Have him create a fresh project folder and launch Claude Code:

mkdir $HOME\session2-demo
cd $HOME\session2-demo
claude --dangerously-skip-permissions

Explain what --dangerously-skip-permissions does: "Normally Claude Code asks permission before every file write, every command, every action. That's good for untrusted environments. On your personal machine, for tasks you trust, this flag tells it to just go. You'll use this most of the time for personal work."
This flag is called 'dangerously' on purpose — it's a reminder that you're giving Claude full autonomy. On your own machine, for your own work, that's fine. On a shared server or with sensitive systems, you'd leave it off and approve each action.

Walk through the essential commands live — have him type each one:

Inside Claude Code, have him run each of these:

/help — show all commands. Point out the ones that matter: /compact, /status, /init, /clear
/status — shows context window usage. "This is your fuel gauge. We're going to watch this number all session."
Shift+Tab x2 — toggles plan mode. "In plan mode, Claude thinks and plans but doesn't execute. Useful when you want to review the approach before it starts writing code."
Esc — cancels whatever Claude is currently doing. "Your emergency brake."
exit — leaves Claude Code back to normal terminal.
Still inside Claude Code, run /init. This creates a CLAUDE.md file in the current folder. Open it and look at it together.
This file is the project's memory. Every time you start Claude Code in this folder, it reads CLAUDE.md first — before you say anything. It's like onboarding a new coworker: 'Here's what this project is, here's how we do things, here's what matters.' We'll write a real one for HouseAccount later.
Don't get bogged down in CLAUDE.md content right now. The point is that he knows it exists and what it does. The deep dive on project context is Session 3.
25–40 min The Context Window — Your Most Important Resource

Goal: Make context window management a tangible, practiced skill — not an abstract concept. He should leave this block able to check context usage, compact, and do session handoffs without thinking.

Here's the thing nobody tells you about AI: every conversation has a shelf life. The context window is Claude's working memory — everything you've said, everything Claude has said, every file it's read, every tool call. It all takes up space. When it fills up, Claude doesn't crash — it just gets worse. It starts forgetting things from earlier in the conversation, missing details, giving generic answers when it was being specific before. This is the number one thing that trips up new operators. So we're going to learn to manage it.

Exercise 1: Watch the context fill in Claude Code

In the Claude Code session from Block 2, have him do a few things — create a file, edit it, ask Claude a question about it. Then run /status. Show him the context usage percentage.
See that number? That's how full your working memory is. Now watch what happens when we read something big.
Have Claude Code read a large file — if he has a contract or long document on his machine, use that. Otherwise, ask Claude to generate a 200-line file and then read it back. Run /status again.
One big file just jumped the number by 10–15%. Now imagine a 90-minute session where you're reading files, writing code, iterating back and forth. You hit 80% faster than you think.

Exercise 2: /compact in action

With the context partially full, run /compact. Have him watch what happens. Then run /status again to see the before/after.
Compact summarizes everything that happened and throws away the raw conversation. You lose the exact words but keep the meaning. The rules of thumb: check /status periodically, compact at 50%, start thinking about a new session at 75%.

Exercise 3: Session handoffs in Chat

Switch to Claude Desktop Chat. Open a conversation — can be an existing one or start a quick one. After a few exchanges, ask Claude:

"Summarize this entire conversation — what we discussed, what was decided, what's still open, and any context I'd need to continue this work in a new session. Format it as a session handoff I can paste into a fresh chat."

Show him the output. Then start a NEW chat, paste the handoff, and continue. Show him that Claude picks up right where it left off.
This is how you work on long projects without degradation. Fresh context window, but Claude knows everything from the last session. Do this every time a conversation gets long or when you're switching between work streams. The handoff takes 30 seconds and saves you from fighting a confused model.

Exercise 4: Recognizing the symptoms

I want you to know what context degradation looks like so you can catch it before it gets bad. The warning signs: Claude starts repeating itself. It forgets instructions you gave earlier. It gives generic responses when it was being specific before. It contradicts something it said 20 messages ago. When you notice any of these — you're past the point where you should have compacted or started fresh.

The rules of thumb to internalize:

  • In Claude Code: /status is your fuel gauge. Compact at 50%. New session at 75%.
  • In Chat: start a new conversation after 30–40 exchanges, or when it feels sluggish. Always do a handoff if the work needs to continue.
  • In Projects: the project instructions persist automatically, but the conversation still fills up. New chat inside the same project = fresh context + same project memory.
  • Never paste an entire long document if you can paste just the relevant section. Context is expensive real estate.
  • claude --resume in the terminal picks up your last Claude Code session — useful when you closed the terminal but want to continue.
Don't rush this block. Context management is more important than any tool he builds today. If he leaves Session 2 with the habit of checking /status and doing handoffs, that's a bigger win than building the perfect app.

Open the HouseAccount Builder project in Claude Desktop Chat. Have Matt describe the tool he wants to Claude. Coach him on the planning prompt:

"I want to build a [tool]. Here's what it needs to do: [requirements]. The user is [who]. Help me think through the inputs, outputs, edge cases, and layout. Then write me the exact prompt I should paste into Claude Code to build this as a single-page web app."

Let Claude do the planning. Review the output together. This is where Matt's domain expertise matters — he knows what fields a quote needs, what job types exist, what zip codes HouseAccount serves.
See what just happened? You described a business problem. Claude turned it into a technical spec. You didn't need to know how to code — you needed to know your business. That's the operator skill: translation.

Phase 2: Build in Claude Code (10 min)

Have Matt create a project folder, launch Claude Code, and paste the prompt from Phase 1:

mkdir $HOME\houseaccount-tool
cd $HOME\houseaccount-tool
claude --dangerously-skip-permissions

Then paste the build prompt. Let Claude Code run. Don't interrupt. Let him watch it plan, write files, and open the result in the browser.

This should take about 60–90 seconds. The working app appearing in his browser from a pasted paragraph is the moment.

That's a working prototype. From a paragraph. No developer, no sprint planning, no two-week estimate. You described what you needed, and it exists now.

Phase 3: Iterate + Context Management (10 min)

Now iterate. Have Matt ask for changes — more features, better design, edge case handling. Encourage 3–4 rounds of iteration:

"Add a field for square footage"
"Make the results color-coded — green for under average, red for over"
"Add a reset button and make it work on mobile"
"Add an export button that copies the result to clipboard"

After each round, have him check /status. When it crosses 50%, have him /compact. If it approaches 75%, do a session handoff to a fresh Claude Code session.
Notice what just happened — you went through the full cycle. Plan in Chat, build in Code, iterate until it's right, manage context along the way. This is the workflow for everything from here on.
Don't over-polish the tool. The point is the workflow, not the app. 3–4 iterations is enough. If he wants to keep going, that's homework.
60–70 min The Batch Mindset + Homework

Goal: Plant the batch thinking seed deeper, and give him concrete homework that builds on today's skills.

One more concept before we wrap. You just built a tool that handles one quote. How many quotes does your team review in a week?

Whatever the number is — 10, 50, 100 — the follow-up is:

What if you pasted a CSV with all 50 and got a table back in 10 seconds? That's the batch mindset. You built the tool for one. The jump to many is trivial once the logic exists. Start noticing this pattern in your week: every time you do something one at a time, ask yourself 'do I have 10 more of these?'
