Homework: Pipeline or Agent?

Session 5 follow-up — due before Session 6
CONNECT Run the Agent on Real Data

The session used sample data designed to illustrate the concepts. Now point it at reality.

SWAP THE DATA
Export your actual signed providers from Metabase. Match the fields to the sample format — name, hourly rate, specialties, service zips, rejection metrics, response time, fulfillment rate. Save as sample_data/real_providers.json (or overwrite providers.json).
RUN BOTH VERSIONS
Run the pipeline and the agent on your real data. Compare the reports. Does the agent catch things the pipeline misses? Are there providers where the agent's judgment is clearly better than the formula?
WHAT TO NOTICE
With real data, the agent's behavior will be less dramatic than the sample — real data is messier and more ambiguous. That's the point. Watch for cases where the agent hesitates, uses more tools, or expresses lower confidence. Those are the genuinely hard cases where the pipeline's fixed logic would get it wrong.
MAP Label Everything You've Built

Go through every AI-powered system or workflow you've built or planned for HouseAccount. For each one, answer one question: who decides the control flow — you or the model?

MAKE THE LIST
Write it out — a simple table or list:
[System/workflow] → Pipeline or Agent → Why Customer intake chatbot → Pipeline → I hardcoded the sequence Provider health outreach → Agent → I don't know the best action per provider yet Rate book matching → Pipeline → Deterministic fuzzy match ...
WHAT YOU'LL FIND
Most of your systems are pipelines. That's correct. The exercise isn't to turn everything into agents — it's to see clearly which problems are solved (pipeline) and which are still being figured out (agent).
HUNT Find the Repeating Patterns

Run the agent 3 more times on different subsets of providers (or the full set). Read the reports each time.

LOOK FOR REPETITION
Which tool sequences show up every time? When the agent sees a pricing problem, does it always do the same thing? When it sees a healthy provider, does it always use the same two tools? Those are your candidate pipelines — the agent is telling you what the fixed sequence should be.
WRITE IT DOWN
For any pattern you spot, write the sequence out: "When [condition], do [step 1] → [step 2] → [step 3]." That's a pipeline definition extracted from agent behavior. This is the collapse in action.
This is the real skill: not building agents or building pipelines — but knowing when to use which, and knowing how to graduate from one to the other as the problem becomes clearer.
BUILD Pick One New Agent

Find something at HouseAccount where you don't yet know the right sequence. Something where the problem isn't legible yet.

SCOPE IT
Write a CLAUDE.md and tools.md for it — same structure as the Provider Health agent. Define the goal, the available tools, and the constraints. Don't define the sequence. That's the whole point — let the agent figure it out.
  • Customer churn analysis — why do homeowners use HouseAccount once and not return?
  • Job categorization — when a homeowner describes a vague problem, which service category fits?
  • Provider recommendation — given a specific job, which provider is the best match (beyond just "can they do it")?
  • Group chat monitoring — interpreting the provider-customer text thread to detect fulfillment or problems
RUN IT AND OBSERVE
Don't try to make it perfect. Run it on a few examples and watch what it does. The goal isn't a production system — it's discovery. What sequence does the agent converge on? That's your future pipeline.
DAILY This Week's Reps
  • 3 real tasks per day in Claude. Keep building. The build reflex doesn't stop because we're thinking about architecture.
  • When you build something new, ask: pipeline or agent? Just the label. Build the awareness.
  • Keep routing interesting content through Claude. The research reflex from Session 4 stacks with this — research is a pipeline (fixed extraction schema), but the decision of what to research next could be an agent.
The shift this week: You're not just building tools anymore. You're thinking about which problems are solved and which are still being figured out. Solved problems get pipelines. Unsolved problems get agents. Agents eventually produce pipelines. That's the whole game.
HELP If Something Goes Wrong

"The agent does the same thing every time" — Your providers might be too similar, or your tools might be too narrow. Add more tools that give the agent more options. Or try a more diverse dataset.

"I can't get my Metabase data into the right format" — Export as CSV, then ask Claude to transform it into the providers.json format. Give it a sample of the target format and a sample of your export. This is a one-shot pipeline task — Claude fills in the blanks.

"I don't know what should be an agent vs. pipeline" — Ask yourself: can I write out the steps right now? If yes, pipeline. If you'd have to say "it depends" at any step, that "it depends" is where the agent lives.

"The agent makes bad decisions" — That's expected in Phase 1. The agent is exploring. Bad decisions with good logging are valuable — they tell you what the constraints should be. Add those constraints to the agent.md or tools.md and run again.
