What This Session Is About

You built a production prototype. It works. Real users are on it. That's a significant milestone — and it shows the build reflex from Session 2 is working.

This session asks the harder question: is what you built durable? If you need to add a major feature in three weeks, how much setup will you have to redo? If a teammate needs to extend it, can they? If you want to build a second prototype with the same architecture, how much of the process would you have to figure out again?

We'll start by reviewing your prototype together, then stress-test its context infrastructure, and finish by packaging what you learned into a reusable skill — a set of instructions that Claude can execute on demand, so the next build goes faster.

15–30 min The Context Audit

We'll open the project together and evaluate its context infrastructure — the layers of information that help Claude (and future-you) understand what this project is and how it works.

There are four layers of context, from most persistent to most temporary. The higher the layer, the less you repeat yourself.

Layer 1 — CLAUDE.md (project memory): Lives in the project folder. Claude reads it automatically on launch. Contains what this project is, how it works, architecture decisions, key patterns.

Layer 2 — Project Instructions (Claude Desktop): Persistent instructions inside a Claude Desktop Project. Contains business context — who you are, what the company does, terminology, preferences.

Layer 3 — Session Handoffs (between conversations): The bridge when a conversation fills up. Contains what was discussed, decided, built, and what's still open.

Layer 4 — In-Conversation Context (temporary): What you paste, upload, or discuss in a single chat. Dies when the conversation ends.

Most people only use Layer 4. AI-native operators build all four.

We'll check each layer against your Railway project:

  • Does a CLAUDE.md exist? Is it comprehensive or a bare skeleton?
  • Do you have a Claude Desktop Project for this build? What's in the instructions?
  • Are key decisions documented — why Railway? Why Metabase? Why this OAuth setup?
  • If a fresh Claude session opened this folder, what would it already know vs. what would it miss?
We'll improve the CLAUDE.md together in real-time — you describe the project, we'll structure it together. This typically takes 5–10 minutes and has an outsized impact on every future session.
30–50 min Cold-Start Tests

Three tests that make reproducibility concrete. Each simulates a real situation and reveals specific gaps to fix.

Test 1: "Future You"

Exit your current Claude Code session. Launch a completely fresh one in the same project folder — no preamble, no setup. Ask Claude to describe the project and add a small feature.

If the CLAUDE.md is solid, Claude orients itself immediately. If not, we see exactly where the gaps are and fix them on the spot. This simulates coming back to a project after a few weeks away.

Test 2: "The Teammate"

If you hired someone tomorrow and said "extend this prototype" — what would they need to know that isn't obvious from the code?

  • How to run the project locally
  • How to deploy (the Railway process)
  • Where credentials and environment variables live
  • What the data model looks like
  • What's intentional architecture vs. a quick fix that should be cleaned up

Answering this question IS the documentation process. We'll capture the answers directly in the CLAUDE.md or a README.

Test 3: "The Next Build"

You connected OAuth, Metabase, and the Anthropic API to a Railway deployment. If you needed to spin up a second prototype with the same stack but different functionality — how much of that setup process would you have to figure out again?

Whatever is reusable gets extracted into a skill in the next block.

The pattern: every test failure is a gap in context infrastructure. Every gap we fix makes the project more robust — and it's not extra work, it's capturing work that already happened.
50–65 min Build a Skill

A skill is a set of instructions — written in plain language — that tells Claude how to do something repeatable. Think of it as a recipe. Once written, you or anyone on your team can hand it to Claude and get a consistent result every time.

Skills are the natural output of the reproducibility mindset: once you've identified something reusable, you package it so it never has to be figured out again.

Skill format:

# Skill: [Name] ## When to use [The trigger — when would you reach for this?] ## Inputs needed - [What you need before starting] - [Credentials, project name, data files, etc] ## Steps 1. [First step — detailed enough for Claude to execute] 2. [Second step] 3. [...] ## Expected output [What the result should look like when done] ## Gotchas - [Common failure and how to handle it] - [Edge case to watch for]
We'll build one together from your Railway prototype process. You walk through what you did from zero to deployed, step by step, and we'll format it as a skill document — a single markdown file that someone could hand to Claude Code and say "set up a new prototype using this pattern."

Where skills can live:

  • In a project folder: Save skill files in a /skills or /docs directory. Reference them from CLAUDE.md so Claude knows they're available.
  • In Project Instructions: For smaller skills, paste them directly into Claude Desktop Project instructions — always in context.
  • As standalone files: Save in a shared Drive folder or a dedicated skills directory. Copy-paste into conversations when needed.

There's no single right answer for where to store them yet. The important thing is that the skill exists as a file, not as a memory.

REF Quick Reference

The Context Stack (most persistent → most temporary):

  • Layer 1 — CLAUDE.md: Project memory. Auto-read on launch. What this project is, how it works, key decisions.
  • Layer 2 — Project Instructions: Business context in Claude Desktop. Who you are, what the company does, terminology.
  • Layer 3 — Session Handoffs: Bridge between conversations. What was discussed, decided, built, what's open.
  • Layer 4 — In-Conversation: What you paste or upload in a single chat. Temporary.

The Three Cold-Start Tests:

  • Future You: Fresh Claude Code session, same folder. Can Claude describe the project and be productive without preamble?
  • The Teammate: What would a new person need to know that isn't in the code? That's the documentation gap.
  • The Next Build: What parts of this process are reusable? That's the skill.
