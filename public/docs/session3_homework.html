CONTEXT Refine Your CLAUDE.md

We improved your Railway project's CLAUDE.md during the session. This assignment is about finishing the job — making it comprehensive enough that a fresh Claude session can be productive immediately.

STEP 1 — REVIEW AND EXPAND
Open your Railway project's CLAUDE.md. Read through what we wrote on the call. Add anything that's missing — architecture decisions, service connections, environment variables, deployment steps, data model, known limitations. Spend 15 minutes. It doesn't need to be perfect, it needs to be useful.
STEP 2 — RUN THE "FUTURE YOU" TEST
Close any existing Claude Code sessions in this project. Open a completely fresh one:
cd [your-railway-project-folder] claude --dangerously-skip-permissions
Then ask Claude — with no other context:
Describe this project. What does it do, what's the tech stack, and how is it deployed?
Compare Claude's answer to reality. Every mistake or gap is something to add to the CLAUDE.md. Fix it, then run the test again until Claude gets it right.
WHY THIS MATTERS
Every minute you spend on the CLAUDE.md saves you five minutes of re-explaining things in future sessions. It's the highest-leverage 15 minutes you can spend on a project. And it compounds — as the project grows, the CLAUDE.md keeps every new conversation starting from a higher baseline.
SKILL Write a Second Skill

We built one skill together on the call — your Railway deployment pattern. Now write a second one on your own. Pick any process you've done more than once.

STEP 1 — PICK YOUR PROCESS
Look for something repeatable in your week. Some ideas:
  • A provider outreach or follow-up workflow
  • How you set up a new Claude Desktop Project with HouseAccount context
  • A data analysis pattern you've used in Metabase or with Claude
  • A customer intake process — how a homeowner request becomes a provider brief
  • How you debug and iterate on a Claude Code build
It doesn't have to be technical. Any process that lives in your head and could be written down counts.
STEP 2 — WRITE IT AS A SKILL FILE
Use the format from the session. Create a new markdown file — in your project folder, on your Desktop, wherever works. The template:
# Skill: [Name] ## When to use [The trigger — when would you reach for this?] ## Inputs needed - [What you need before starting] - [Credentials, project name, data files, etc] ## Steps 1. [First step — detailed enough for Claude to execute] 2. [Second step] 3. [...] ## Expected output [What the result should look like when done] ## Gotchas - [Common failure and how to handle it] - [Edge case to watch for]
STEP 3 — TEST IT
Hand the skill to Claude — either paste it into a Chat conversation or into Claude Code. Ask Claude to execute it (or walk through it step by step). Does it make sense? Does Claude ask clarifying questions that reveal gaps in the instructions? If so, update the skill.
The bar is low on purpose. A rough skill that exists is better than a perfect template that doesn't. Your first few skills will be imperfect. That's fine — they improve with use, just like everything else in this process.

REF Quick Reference

The Context Stack:

  • Layer 1 — CLAUDE.md: Project memory. Auto-read on launch.
  • Layer 2 — Project Instructions: Business context in Claude Desktop.
  • Layer 3 — Session Handoffs: Bridge between conversations.
  • Layer 4 — In-Conversation: What you paste or upload. Temporary.

The Three Cold-Start Tests:

  • Future You: Fresh session, same folder. Can Claude orient itself?
  • The Teammate: What would a new person need that isn't in the code?
  • The Next Build: What's reusable? That's a skill.

Skill Template:

# Skill: [Name] ## When to use ## Inputs needed ## Steps ## Expected output ## Gotchas

Commands:

  • /status — context window usage
  • /compact — compress conversation
  • /init — generate CLAUDE.md
  • claude --dangerously-skip-permissions — auto-approve mode
  • claude --resume — pick up last session
HELP If Something Goes Wrong

"Claude doesn't understand my project even with a CLAUDE.md" — Check that the file is named exactly CLAUDE.md (case-sensitive) and is in the root of the project folder. Run /init if you're not sure — it will either create one or tell you one exists.

"I don't know what to put in the CLAUDE.md" — Start by telling Claude Code: "Look at this project and write a CLAUDE.md that describes what it is, the tech stack, how to run it, and how to deploy it." Review the output and add anything it missed — especially decisions and context that aren't obvious from the code.

"My skill doesn't work when I give it to Claude" — The steps are probably too vague. Claude needs specifics — exact commands, file names, configuration values. If Claude asks a clarifying question when running the skill, the answer to that question should be added to the skill.

"I'm not sure what counts as a skill" — If you've explained it to Claude more than once, it's a skill. If you've copy-pasted the same setup instructions into multiple conversations, it's a skill. If a new teammate would ask you "how do I do X?" — the answer is a skill.

Anything else breaks — Screenshot the error, paste it into Claude Chat, and say: "I'm trying to [what you were doing] and hit this. What should I do?" Or drop it in the Jottings tab and we'll troubleshoot at the start of Session 4.
