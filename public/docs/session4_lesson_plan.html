What This Session Is About

Everything you've built so far has been inward-facing — your data, your providers, your rate book. Claude has been a builder. That's powerful, but it's half the picture.

This session introduces the other half: using Claude to go outward. But we're not just going to Google some articles and ask for summaries. We're going to build a research engine — a directory of markdown files that turns Claude Code into a reusable tool you can point at any topic and get structured, actionable intelligence back.

The inspiration comes from Andrej Karpathy's "auto-research" concept — using AI to systematically fetch, process, and synthesize information from multiple sources into a coherent report. We're going to take that concept, strip it to what matters, and build a version of it that lives in your toolkit permanently. You'll be able to point it at pricing research, competitor analysis, market intelligence, provider data — anything.

0–15 min The Shift: Builder → Researcher

Quick check-in on what's happened since last session — what you built, what broke, what surprised you. Then the reframe:

You've been using Claude to build things — a chat interface, a pricing engine, an admin dashboard. Every time you open Claude, you come with a task. That's the builder reflex, and it's working.

But think about how you consume information right now. You read an article, listen to a podcast, skim a competitor's site. Some of it sticks, most doesn't. What if every interesting thing you encountered got routed through Claude and came out the other side as something structured, searchable, and useful?

The research reflex: when you encounter something interesting — an article, a podcast, a competitor's pricing page, a market report — your first instinct is to feed it to Claude and ask "what can I do with this?" Not read it. Process it.

That's the one-at-a-time version. Today we're going to build the system version — a research engine that does this at scale, on any topic, repeatable and improvable.

The concept comes from Andrej Karpathy — one of the original architects behind OpenAI, Tesla Autopilot, and a bunch of foundational AI work. He published a project called "auto-research" that uses Claude to systematically research a topic: define the question, fetch sources, extract findings, synthesize a report. We're going to take his architecture, simplify it, and build a version you own.

15–35 min The Architecture: Markdown Files as a Machine

Here's the core idea: a directory of markdown files IS a tool. Claude Code reads the directory, follows the instructions in the .md files, and produces structured output. No app to build. No code to write. Just well-written instructions in the right structure.

The anatomy of a research engine:

research-engine/ ├── CLAUDE.md # The brain — what this tool does, how to run it ├── methodology.md # How to research — source types, search patterns, depth ├── extraction.md # What to extract — fields, format, structure ├── synthesis.md # How to combine findings into a final output ├── topics/ # One .md per research topic │ ├── pricing.md # "Research home services pricing in our markets" │ └── competitors.md # "Map the competitive landscape" └── output/ # Where finished reports land └── .gitkeep

Each file has a job:

  • CLAUDE.md — the entry point. When you open this directory in Claude Code, it reads this first. It explains what the tool does and how to invoke it. "I want to research [topic]" triggers the whole workflow.
  • methodology.md — defines how Claude should research. What kinds of sources to look for, how many, how deep to go, what to prioritize (primary sources over aggregators, recent over old, data over opinion).
  • extraction.md — defines the output schema. What fields to pull from every source. This is what turns a messy article into a structured data point you can compare across sources.
  • synthesis.md — defines how to combine individual extractions into a final report. The structure, the format, the level of analysis.
  • topics/ — one file per research question. Each file frames the specific question, provides business context, and lists any known sources to start from.
This is the "skills" concept from Session 3, scaled up. A skill is one .md file that tells Claude how to do one thing. A research engine is a directory of .md files that tell Claude how to do something complex — with separation of concerns. The methodology doesn't change when you change the topic. The extraction schema doesn't change when you change the methodology. Each piece is independent and reusable.

The model split: When we built our version, we used Opus to design the .md files — the methodology, the extraction schema, the synthesis logic. Then we used Sonnet to run the actual research sessions. Opus is the architect; Sonnet is the researcher. This keeps costs down and quality up — you don't need the most expensive model for fetching and extracting, but you want it for designing the system.

Why this matters beyond research: Once you understand that a directory of .md files is a tool, you'll start seeing it everywhere. Onboarding processes, sales playbooks, content generation workflows, data analysis pipelines — any repeatable process can be structured this way and handed to Claude Code. The research engine is the first one you build, not the last.
35–55 min Build It: Your HouseAccount Research Engine

We're going to build this live. Not a demo — the actual tool you'll use going forward.

Create a new directory on your machine — something like houseaccount-research. We'll build the .md files together, one at a time, talking through what each one should say before writing it.

Step 1: The CLAUDE.md

This is the file Claude Code reads first when you open this directory. We'll write it to explain what the tool does and how to invoke it. Something like: "This is a research engine for HouseAccount. When given a topic, follow the methodology in methodology.md, extract findings using the schema in extraction.md, and synthesize a report using synthesis.md."

Step 2: The methodology

How should Claude research? What sources are credible for home services intelligence? We'll define tiers: primary sources (competitor websites, industry reports, franchise filings, government data) over secondary (blog posts, news articles) over tertiary (forums, social media). How many sources is enough? How recent should they be?

Step 3: The extraction schema

What do you want from every source? For HouseAccount, this might include: company or source name, date, service categories mentioned, pricing data points, geographic coverage, provider acquisition approach, technology or AI usage, customer acquisition strategy, fulfillment model. Consistent fields across every source means you can compare apples to apples.

Step 4: The synthesis template

How should Claude combine findings? Executive summary, key findings by category, data table, recommendations, gaps to investigate further. This is the output format for every report the engine produces.

Step 5: The first topic file

We'll pick a real research question for HouseAccount and write the topic file together. This frames the specific question, provides business context (what HouseAccount does, what you already know, what decisions this research would inform), and optionally lists starting sources.

The meta-lesson: We're using Claude to build a tool made of instructions for Claude. The .md files are the product. Claude Code is the engine that runs them. You don't need to learn a programming language or build an app — you need to learn how to write clear, structured instructions. You've been doing that since Session 1.
55–70 min Run It: Point and Fire

Time to test. We'll open the research engine directory in Claude Code and give it the topic we just wrote.

Open Claude Code in the houseaccount-research directory. Tell it: "Research the topic in topics/[your-topic].md — follow the methodology, extract using the schema, and produce a synthesis report in output/." Watch it work.

What to watch for:

  • Does Claude follow the methodology? Is it finding the right kinds of sources?
  • Is the extraction schema working? Are the fields being populated consistently?
  • Is the synthesis useful? Would you actually make a decision based on this report?
  • Where does it break? Every gap is a chance to improve the .md files.

Iterate the system, not the output. If the report is too shallow, you don't rewrite the report — you improve the methodology or the extraction schema. If it's missing a category of information, you add it to the schema. The system gets better every time you run it.

This is the repetition-to-automation ladder from the mantras, applied to research. First you do it manually (Session 3 skills). Then you build a system (this session). The system improves with use. Eventually, you schedule it — a weekly competitive intelligence brief that runs itself. But the manual reps come first.

The versatility test: If time allows, we'll write a second topic file on a completely different subject — maybe provider pricing in a new market, or AI adoption in home services. Same engine, different input. The output should be equally useful. That's how you know the architecture works.

70–75 min Homework & What's Next

Homework (due before Session 5):

  • Run the engine on 2 more topics. Write two new topic files for research questions that matter to HouseAccount right now — pricing in a new market, provider retention strategies, customer acquisition costs in home services, whatever's top of mind. Run each one and save the output.
  • Improve the system. After running it a few times, you'll notice what's working and what's not. Update the methodology, extraction schema, or synthesis template based on what you learn. The .md files should be better at the end of the week than they are today.
  • Try it on something non-HouseAccount. Point the engine at a personal interest — an investment thesis, a hobby, a technology you're curious about. Write a topic file, run it, see what comes back. This tests whether the architecture is truly general-purpose.
  • Feed it a thought leader. Find a podcast episode, a YouTube video, or a long blog post from someone whose thinking you respect. Instead of reading or watching it, feed the transcript to Claude and extract the operational ideas using your research engine's schema. Compare what you get vs. what you would have absorbed passively.
  • Daily reps + new habits. Keep building on the prototype. Try the screenshot-paste-into-Code pattern for bugs. Try Plan mode (Shift+Tab twice) before big changes. When you read something interesting, process it — don't just skim.

REF Quick Reference

The Research Engine Architecture:

research-engine/ ├── CLAUDE.md # Entry point — what this tool does, how to run it ├── methodology.md # How to research — sources, depth, priorities ├── extraction.md # What to extract — consistent fields per source ├── synthesis.md # How to combine findings into a report ├── topics/ # One .md per research question └── output/ # Finished reports

The Model Split:

  • Opus designs the system — methodology, extraction schema, synthesis logic. The architect.
  • Sonnet runs the sessions — fetching, extracting, producing reports. The researcher.

Three Levels of Research:

  • Level 1 — One-shot: Feed Claude a source, get a summary. Useful but doesn't compound.
  • Level 2 — Structured extraction: Extract into consistent fields. Now it's data you can compare.
  • Level 3 — Research engine: A directory of .md files that does Level 2 at scale, on any topic, repeatably.

Source Types Claude Can Process:

  • URLs (articles, blog posts, landing pages, pricing pages)
  • PDFs (reports, whitepapers, franchise filings)
  • Transcripts (podcasts, YouTube videos, meeting recordings)
  • Screenshots (competitor dashboards, error messages, UI patterns)
  • Raw text (forum posts, email threads, Slack exports)
